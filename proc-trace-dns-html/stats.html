Stats & Logging

Pipe proc-trace-dns into proc-trace-logger to record every DNS event to SQLite — then query history, spot patterns, and audit traffic over time.

SETUP

Running the logger

proc-trace-logger reads proc-trace-dns -j -t output from stdin and writes batched inserts to a SQLite database. Run both in the background — the pipe keeps them connected.

Tip: always pass -t to proc-trace-dns when logging. Without it, the JSON has no timestamp and the logger falls back to insertion time, which is slightly less accurate.
start logging in the background 
# Start capturing — pipe JSON output into the logger
$ sudo proc-trace-dns -j -t | proc-trace-logger &

# Or specify a custom DB path
$ sudo proc-trace-dns -j -t | proc-trace-logger --db /data/dns.db &

# Quiet mode (suppress progress messages)
$ sudo proc-trace-dns -j -t | proc-trace-logger -q &

# Stop logging
$ kill %1
proc-trace-logger flags 
proc-trace-logger [flags]

  --db       PATH     SQLite database path  (default: /var/lib/proc-trace/dns.db)
  --batch    N        rows per INSERT transaction  (default: 50)
  --flush-ms N        max ms to hold partial batch  (default: 500)
  -q                  suppress progress output
SCHEMA

Database schema

One row per DNS response. Indexed on ts, query, name, and rcode for fast filtering.

dns_events 
CREATE TABLE dns_events (
    id          INTEGER  PRIMARY KEY AUTOINCREMENT,
    ts          INTEGER  NOT NULL,   -- Unix milliseconds (UTC)
    pid         INTEGER  NOT NULL,   -- process ID
    name        TEXT     NOT NULL,   -- process name (e.g. "curl")
    type        TEXT     NOT NULL,   -- record type: A, AAAA, MX, TXT, …
    query       TEXT     NOT NULL,   -- hostname queried
    answers     TEXT     NOT NULL,   -- JSON array of resolved addresses
    rcode       TEXT     NOT NULL,   -- NOERROR | NXDOMAIN | SERVFAIL | …
    latency_ms  REAL     NOT NULL    -- query round-trip time in ms
);
ColumnTypeNotes
idINTEGERAuto-increment primary key
tsINTEGEREvent time as Unix milliseconds. Use datetime(ts/1000,'unixepoch') to format
pidINTEGERPID of the process that made the query
nameTEXTProcess name (from /proc/<pid>/comm)
typeTEXTDNS record type: A, AAAA, CNAME, MX, TXT, PTR, SRV, NS
queryTEXTHostname that was queried
answersTEXTJSON array of resolved values, e.g. ["1.2.3.4","5.6.7.8"]
rcodeTEXTDNS response code: NOERROR, NXDOMAIN, SERVFAIL, REFUSED, TIMEOUT
latency_msREALRound-trip query time in milliseconds
QUERIES

Useful sqlite3 queries

Open the database with sqlite3 /var/lib/proc-trace/dns.db and run any of the following.

Database overview

Total events captured, time range, and current database size.

overview
SELECT
    COUNT(*) AS total_events,
    datetime(MIN(ts)/1000, 'unixepoch') AS oldest,
    datetime(MAX(ts)/1000, 'unixepoch') AS newest,
    ROUND(AVG(latency_ms), 2) AS avg_latency_ms
FROM dns_events;

Top queried domains

Which hostnames are being resolved most? Change the time filter to narrow the window.

top domains — last hour
SELECT query, COUNT(*) AS cnt
FROM   dns_events
WHERE  ts > (strftime('%s', 'now') - 3600) * 1000
GROUP BY query
ORDER BY cnt DESC
LIMIT 20;

Top processes by query count

Which processes are generating the most DNS traffic?

busiest processes
SELECT name, COUNT(*) AS total, COUNT(DISTINCT query) AS unique_domains
FROM   dns_events
GROUP BY name
ORDER BY total DESC
LIMIT 15;

NXDOMAIN failures

Find failed lookups — misconfigured services, typos, or suspicious probing.

NXDOMAIN failures
SELECT
    datetime(ts/1000, 'unixepoch') AS time,
    name, query, latency_ms
FROM   dns_events
WHERE  rcode = 'NXDOMAIN'
ORDER BY ts DESC
LIMIT 50;

Slowest queries

High latency lookups indicate slow upstream resolvers or network issues.

slowest queries
SELECT name, query, type, ROUND(latency_ms, 2) AS latency_ms
FROM   dns_events
ORDER BY latency_ms DESC
LIMIT 20;

Query volume over time

Traffic histogram by hour for the last 24 hours — spot spikes and quiet periods.

hourly volume — last 24h
SELECT
    strftime('%Y-%m-%d %H:00', datetime(ts/1000, 'unixepoch')) AS hour,
    COUNT(*) AS count
FROM   dns_events
WHERE  ts > (strftime('%s', 'now') - 86400) * 1000
GROUP BY hour
ORDER BY hour;

What does a process resolve?

See every unique domain a specific process has queried — useful for dependency auditing.

domains per process
SELECT DISTINCT query, type, rcode
FROM   dns_events
WHERE  name = 'curl'          -- replace with your process
ORDER BY query;

Detect beaconing

Processes that repeatedly resolve the same domain at regular intervals — a beaconing signature.

repeated lookups by process + domain
SELECT name, query, COUNT(*) AS count,
       ROUND(AVG(latency_ms), 2) AS avg_ms
FROM   dns_events
GROUP BY name, query
HAVING count > 10
ORDER BY count DESC
LIMIT 30;

Latency percentiles

P50 / P95 / P99 resolver latency — requires SQLite 3.41+ for percentile() or use the manual approach below.

latency distribution
SELECT
    ROUND(MIN(latency_ms), 2)  AS min_ms,
    ROUND(AVG(latency_ms), 2)  AS avg_ms,
    ROUND(MAX(latency_ms), 2)  AS max_ms
FROM   dns_events
WHERE  rcode = 'NOERROR';

-- P95 via row ordering
SELECT ROUND(latency_ms, 2) AS p95_ms
FROM   dns_events
WHERE  rcode = 'NOERROR'
ORDER BY latency_ms
LIMIT  1
OFFSET (SELECT CAST(COUNT(*) * 0.95 AS INTEGER) FROM dns_events WHERE rcode = 'NOERROR');

Record type breakdown

Distribution of A, AAAA, MX, TXT, and other query types across all captured traffic.

record type distribution
SELECT type, COUNT(*) AS count,
       ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM dns_events), 1) AS pct
FROM   dns_events
GROUP BY type
ORDER BY count DESC;
MAINTENANCE

Managing the database

On a busy system the database can grow quickly. Use these commands to keep it under control.

housekeeping
# Check current size
$ du -sh /var/lib/proc-trace/dns.db

# Delete events older than 7 days
$ sqlite3 /var/lib/proc-trace/dns.db \
    "DELETE FROM dns_events WHERE ts < (strftime('%s','now') - 604800) * 1000"

# Reclaim freed pages
$ sqlite3 /var/lib/proc-trace/dns.db "VACUUM"

# Row count and WAL status
$ sqlite3 /var/lib/proc-trace/dns.db "SELECT COUNT(*) FROM dns_events; PRAGMA wal_checkpoint;"
