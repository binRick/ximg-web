Unix Philosophy

Formulated by Doug McIlroy and articulated by Ken Thompson, the Unix philosophy is a set of design principles that made Unix software composable, maintainable, and powerful. Eric Raymond later distilled it into 17 rules in The Art of Unix Programming.

The Three Core Rules (McIlroy)

  • RULE 1
    Write programs that do one thing and do it well.
    Each tool has a single, clear purpose. ls lists. grep filters. wc counts. Nothing more.
  • RULE 2
    Write programs to work together.
    Tools are designed to be composed. The output of one becomes the input of the next via pipes.
  • RULE 3
    Write programs to handle text streams, because that is a universal interface.
    Plain text is human-readable, scriptable, and works across every tool, language, and protocol.

Eric Raymond's 17 Rules

  • Rule of Modularity
    Build simple parts connected by clean interfaces.
  • Rule of Clarity
    Clarity is better than cleverness.
  • Rule of Composition
    Design programs to be connected with other programs.
  • Rule of Separation
    Separate policy from mechanism; separate interfaces from engines.
  • Rule of Simplicity
    Design for simplicity; add complexity only where you must.
  • Rule of Parsimony
    Write a big program only when nothing else will do.
  • Rule of Transparency
    Design for visibility to make inspection and debugging easier.
  • Rule of Robustness
    Robustness is the child of transparency and simplicity.
  • Rule of Representation
    Fold knowledge into data so program logic can be stupid and robust.
  • Rule of Least Surprise
    In interface design, always do the least surprising thing.
  • Rule of Silence
    When a program has nothing surprising to say, it should say nothing.
  • Rule of Repair
    When you must fail, fail noisily and as soon as possible.
  • Rule of Economy
    Programmer time is expensive; conserve it in preference to machine time.
  • Rule of Generation
    Avoid hand-hacking; write programs to write programs when you can.
  • Rule of Optimization
    Prototype before polishing. Get it working before you optimize it.
  • Rule of Diversity
    Distrust all claims for one true way.
  • Rule of Extensibility
    Design for the future, because it will be here sooner than you think.
