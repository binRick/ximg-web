Nagios Core

Open-source infrastructure monitoring — hosts, services, and alerts since 1999

What is Nagios Core?

Nagios Core is an open-source IT infrastructure monitoring system. It watches hosts and services, alerts on failures, and provides a web dashboard showing the health of your entire environment at a glance. Originally called NetSaint, it was renamed Nagios in 2002 and became the gold standard for ops monitoring for over a decade.

Nagios uses a plugin model — the engine itself just schedules checks and handles alerting. All actual monitoring logic lives in small executables (plugins) that return a numeric exit code: 0 = OK, 1 = WARNING, 2 = CRITICAL, 3 = UNKNOWN. This makes it infinitely extensible.

Architecture

nagios core (scheduler + alert engine)
│
├── check_http → HTTP/HTTPS target → 0/1/2/3
├── check_ping → ICMP target → 0/1/2/3
├── check_tcp → TCP port target → 0/1/2/3
├── check_disk → local filesystem → 0/1/2/3
│
├── check_nrpe → NRPE agent (port 5666) on remote host
│ │
│ └── runs plugins locally (check_disk, check_load, …)
│
└── passive checks ← external systems push results via NSCA/API

State Types

Soft State
A check has failed but the maximum retry count has not been reached. Nagios re-checks at the retry_check_interval rate. No notifications are sent during soft states — this absorbs transient blips.
Hard State
The check has failed max_check_attempts times in a row. Notifications fire, the problem is logged as confirmed, and checks resume at normal_check_interval.
Recovery
After a HARD PROBLEM, the next OK result triggers a recovery event. A recovery notification is sent and the service returns to normal scheduling.
Downtime
Scheduled downtime suppresses alerts for a host or service during planned maintenance. Nagios still checks and records state, but skips notification for the duration.

Active vs Passive Checks

Active Checks
Nagios initiates the check on a schedule. The scheduler calls a plugin, waits for the exit code, and records the result. Most checks are active — HTTPS, ping, TCP port, load, disk, etc.
Passive Checks
External systems push results to Nagios via the command file (FIFO) or NSCA. Useful for services that can't be reached from the Nagios host, batch jobs, or event-driven state changes.

Standard Nagios Plugins (check_*)

check_http -H host -S --sni
HTTPS check with SNI; add -C 30,14 for cert expiry warning/critical days
check_ping -H host -w 100,5% -c 200,10%
ICMP ping — warn at 100ms/5% loss, critical at 200ms/10% loss
check_tcp -H host -p 443
TCP port connectivity check — simple open/closed test
check_smtp -H host -p 25
SMTP banner check on port 25 (or 587 for submission)
check_disk -w 20% -c 10% -p /
Disk usage — warn at <20% free, critical at <10% free
check_load -w 5,3,2 -c 10,6,4
System load averages at 1/5/15 minutes
check_procs -c 1: -C nginx
Count processes matching a name — critical if count <1
check_nrpe -H host -c check_disk
Delegate a check to NRPE agent on remote host
check_dummy 0 "OK"
Always returns specified exit code — useful for passive-only services

NRPE — Nagios Remote Plugin Executor

NRPE lets Nagios run plugins on remote hosts. The agent listens on TCP 5666 and accepts check commands from the Nagios server. This is necessary for local-only checks like disk usage, CPU load, or process counts on machines that don't expose those metrics over the network.

The Nagios server runs check_nrpe -H <target> -c <command>. The NRPE daemon on the target executes the corresponding plugin locally and returns the exit code and output back over the connection.

Obsessing

Obsessive checks run an additional event handler script after every check result, regardless of state change. Combined with obsess_over_services=1 in nagios.cfg, this feeds results to distributed monitoring parents (DistributedNagios / OCSP/OCHP). Used to collect check data from child Nagios nodes into a central aggregator.

Object Definitions

Nagios configuration is built from object definitions in .cfg files under the cfg_dir path. The four core objects are host, service, command, and contact. All share a template/inheritance system via use directives.

Host Definition

# Template — inherited by all hosts define host { name ximg-host-template check_command check_ping!100,5%!200,10% max_check_attempts 3 check_interval 5 # minutes between checks retry_interval 1 # minutes between retries on failure check_period 24x7 notification_interval 60 notification_period 24x7 notifications_enabled 1 register 0 # template, not a real host } define host { use ximg-host-template host_name ximg.app alias ximg landing page address ximg.app contact_groups admins }

Service Definition

# Service checks run against a specific host define service { host_name ximg.app service_description HTTPS check_command check_https_vhost max_check_attempts 3 check_interval 5 retry_interval 1 check_period 24x7 notification_interval 60 notification_period 24x7 contact_groups admins notifications_enabled 1 } define service { host_name ximg.app service_description SSL-Cert check_command check_https_cert # -C 30,14 max_check_attempts 2 check_interval 360 # every 6 hours retry_interval 60 check_period 24x7 contact_groups admins }

Command Definition

# Commands define what plugin binary to run define command { command_name check_https_vhost command_line $USER1$/check_http -H '$HOSTADDRESS$' -S --sni -p 443 -t 15 -w 3 -c 8 } define command { command_name check_https_cert command_line $USER1$/check_http -H '$HOSTADDRESS$' -S --sni -p 443 -C 30,14 -t 15 } # $USER1$ = /opt/nagios/libexec (set in resource.cfg) # $HOSTADDRESS$ = host address field from host definition # $ARG1$ ... $ARG32$ = !-delimited arguments in check_command

Notification Flow

1
Check fails — plugin returns exit code 1 (WARNING) or 2 (CRITICAL). State changes to SOFT problem.
2
Retries — Nagios re-checks at retry_interval. No notifications yet. After max_check_attempts consecutive failures, state becomes HARD.
3
Hard state reached — Nagios evaluates notification conditions: is notifications_enabled? Is the contact's notification period active? Is the service in scheduled downtime? Is this a flapping service?
4
Notification sent — Nagios runs the contact's service_notification_command (e.g., send an email or POST to Slack webhook). Subsequent alerts fire at notification_interval until the problem resolves.
5
Recovery — check returns 0 (OK). If the previous hard state triggered a notification, a recovery notification fires. State clears to OK and normal scheduling resumes.

Flap Detection

Flapping is when a service oscillates rapidly between OK and problem states. Nagios detects it by tracking the percentage of state changes over the last 21 check results. If the change rate exceeds high_flap_threshold (default 30%), notifications are suppressed and a "FLAPPING START" event fires. When the rate drops below low_flap_threshold (default 20%), a "FLAPPING STOP" event fires and normal alerting resumes.

Enable flap detection globally with enable_flap_detection=1 in nagios.cfg, and per-object with flap_detection_enabled 1.

Escalations

# Escalate to on-call after 3 notifications (3 × 60min = 3 hours unresolved) define serviceescalation { host_name * service_description * first_notification 3 last_notification 0 # 0 = unlimited notification_interval 30 # 30 min cadence after escalation contact_groups oncall escalation_options w,u,c,r }

Acknowledgements

Acknowledging a problem in the web UI (or via command) suppresses further notifications until the next state change. A "sticky" acknowledgement persists across state changes. The acknowledgement is recorded in the log with an author and comment, providing an audit trail. Useful for telling Nagios "I know about this — stop paging me."

Common nagios.cfg Tuning

check_result_reaper_frequency=10
How often (seconds) Nagios collects check results from worker threads
max_concurrent_checks=50
Cap parallel checks to avoid CPU spikes; 0 = unlimited
service_check_timeout=60
Seconds before a check plugin is killed; tune per slow targets
log_rotation_method=d
Rotate logs daily (d), weekly (w), monthly (m), or hourly (h)
enable_event_handlers=1
Run event handler scripts on state changes (auto-remediation)
process_performance_data=1
Pass plugin perfdata to PNP4Nagios / Graphite / InfluxDB for graphs

Timeline

1996
Ethan Galstad begins writing a network monitoring tool for internal use at a Minneapolis company. The project starts as a simple host-down alerter.
1999
NetSaint 0.0.1 released publicly on SourceForge. The tool gains traction in the Linux sysadmin community. Galstad renames it Nagios (Nagios Ain't Gonna Insist On Sainthood) after trademark pressure.
2002
Nagios 1.0 released. Plugin API stabilized — any executable returning 0/1/2/3 is a valid check. The community begins building hundreds of plugins. NRPE released for remote host monitoring.
2006
Nagios 2.0 — major rewrite with improved performance, flap detection, and web CGI redesign. NDOUtils released to store Nagios data in MySQL for reporting.
2007
Nagios Enterprises founded. Commercial Nagios XI announced — a web-based configuration wizard and expanded dashboards built on Nagios Core. The split between "Core" (open source) and "XI" (commercial) begins.
2009
Nagios 3.0 — embedded Perl interpreter removed, adaptive monitoring, improved dependency handling, and template-based object caching. Remains the most widely deployed version for years.
2013
Naemon forked from Nagios Core by contributors frustrated with slow upstream development. Nagios Core 4.0 also released, adding parallel check processing and improved performance for large installations.
2015
Icinga 2 (another fork) gains widespread adoption in Europe, offering a modern config DSL and native clustering. Nagios remains dominant in North America.
2019
Nagios Core 4.4.x released with security hardening and fixes. The plugin ecosystem exceeds 4,000 community-contributed plugins via Nagios Exchange.
2024
Nagios Core 4.5.x — continued security patches and maintenance. Still in production at thousands of enterprises globally. Prometheus and Grafana have replaced it for new deployments, but Nagios remains the template that all modern monitoring systems were designed against.
