Show working tree status — staged, unstaged, untracked files

Stage a specific file for the next commit
git add -p

git cherry-pick <hash>

Core Concepts
Commit
A snapshot of the entire working tree at a point in time. Each commit stores a tree hash, parent hash(es), author, committer, timestamp, and message. Commits are immutable — identified by a SHA-1 (now SHA-256 in new repos) hash.
Branch
A lightweight, movable pointer to a commit. Creating a branch costs nothing — just 41 bytes on disk. The default branch is usually called main or master. HEAD points to the currently checked-out branch or commit.
Staging Area
Also called the "index." A preparation area between the working tree and the repo. You choose exactly which changes go into the next commit. This is what makes git add -p so powerful — surgical commits.
Merge
Integrates two branches. A fast-forward merge simply moves the pointer when there's no divergence. A three-way merge creates a merge commit. --no-ff forces a merge commit even when fast-forward is possible.
Rebase
Rewrites commits as if they branched from a different base. Produces a cleaner linear history. Never rebase commits that have been pushed to a shared remote — it rewrites history others depend on.
HEAD
A special pointer that tracks what you have checked out. Normally points to a branch name. In "detached HEAD" state it points directly to a commit hash — useful for exploring history but be careful not to commit there accidentally.
Remote
A named reference to another repository URL (usually origin). Git stores a local copy of each remote branch as a remote-tracking ref like origin/main. These are updated by git fetch.
Tag
A fixed pointer to a specific commit — unlike branches, tags don't move. Lightweight tags are just refs; annotated tags are full Git objects with a message, tagger, and timestamp. Used for release versioning (e.g., v2.4.1).
Stash
A stack of temporary shelves for work-in-progress. Saves tracked modified files and staged changes so you can switch context cleanly. Use git stash push -u to also stash untracked files.
Reflog
A local, time-stamped log of every position HEAD and branch tips have pointed to — even after resets and rebases. Your safety net for recovering "deleted" commits. Entries expire after 90 days by default.
.gitignore
Patterns of files Git should never track. Supports wildcards, negations (!), and directory scoping. Checked-in files are never automatically ignored — use git rm --cached to stop tracking them.
Object Store
Git stores four object types in .git/objects/: blob (file content), tree (directory), commit, and tag. All are content-addressed by their SHA hash — making Git a content-addressable filesystem.
Common Workflows
Centralized Workflow
Everyone commits to main. Simple for small teams. Conflicts resolved on push.
Feature Branch Workflow
Each feature gets its own branch. Merged to main via pull request after review.
Forking Workflow
Contributors fork the repo, push to their fork, open PRs to the upstream. Standard for open source.
Trunk-Based Dev
Short-lived branches (hours to 1-2 days), merge to main frequently. Requires strong CI/CD and feature flags.
GitFlow Strategy
Branch Strategies Compared
GitFlow
Two long-lived branches (main + develop) plus short-lived feature, release, and hotfix branches. Great for versioned software with scheduled releases. Can feel heavyweight for fast-moving teams.
GitHub Flow
Just main + feature branches. Feature branch → PR → review → merge to main → deploy. Simple, fast, suited to continuous deployment. No release branch needed if you deploy from main.
Trunk-Based Dev
All devs commit to main (the "trunk") daily or via very short-lived branches (<2 days). Feature flags hide incomplete features in prod. Maximizes integration frequency, minimizes merge conflicts.
OneFlow
Simplified GitFlow — only one long-lived main branch. Feature branches merge directly to main. Hotfixes branch from a release tag. Cleaner than GitFlow, more disciplined than GitHub Flow.

Merge vs Rebase
git merge
Preserves full history with a merge commit. Non-destructive — existing commits never change. History can look noisy with many merges. Safe on shared/public branches.
git rebase
Rewrites commits onto a new base — clean, linear history. Makes bisect and log easier to follow. Destructive — changes SHAs. Never rebase commits others have pulled.
Squash Merge
Combines all branch commits into one commit on main. Keeps main history clean. Loses granular feature commit history. Popular in PR-based GitHub workflows.
Fast-Forward
When main has no new commits since branch point, Git just moves the pointer — no merge commit. Use --no-ff to force a merge commit and preserve the branch topology.
Golden rule: Never rebase or force-push commits that exist on a remote branch other people are using. Rewriting shared history causes everyone else to diverge and can result in lost work.

Git Timeline
2002
Linux Kernel Adopts BitKeeper
Linus Torvalds and the Linux kernel project begin using BitKeeper, a proprietary distributed VCS, free for open-source use. It works well — but the dependency on a closed tool worries the community.
2005
BitKeeper Revokes Free License Crisis
In April 2005, BitMover revokes the free-use license after Andrew Tridgell reverse-engineers the BitKeeper protocol. Linus Torvalds needs a replacement — fast — for managing the world's largest open-source project.
2005
Linus Writes Git in 10 Days Birth of Git
Torvalds starts writing Git on April 3, 2005. The first self-hosted Git commit is made on April 7 — Git is used to manage its own source code. Designed around speed, correctness, and distributed workflows. The name "git" is British slang for "unpleasant person" — Linus called it a joke.
2005
Linux Kernel Migrates to Git Adoption
By June 2005, the Linux kernel — then 6.7 million lines of code across 27,000 files and 2,000 contributors — is fully migrated to Git. Junio Hamano takes over as maintainer, a role he holds to this day.
2005
Git v1.0 Released v1.0
Git 1.0 released December 21, 2005. Core features: branching, merging, rebasing, diff, stash, log, and remotes. The data model — content-addressable blobs, trees, commits — remains unchanged to this day.
2007
Git Gains Windows Support
msysGit brings native Git to Windows. Previously a Unix-only tool, Git begins reaching a much wider developer audience beyond Linux and macOS.
2008
GitHub Launches Game Changer
Tom Preston-Werner, Chris Wanstrath, and PJ Hyett launch GitHub in April 2008. It adds a social layer to Git — pull requests, forks, issues, and a web UI. Within two years, it becomes the center of open-source development worldwide.
2011
GitLab Founded
Dmitriy Zaporozhets launches GitLab as a self-hosted alternative to GitHub. It later adds CI/CD pipelines, DevSecOps features, and a SaaS offering, becoming a major platform for enterprise teams.
2013
Git v1.8 — Reachability & Performance
The 1.8.x series brings major performance improvements for large repositories, smarter pack operations, and the rerere feature (reuse recorded resolution) for repeated merge conflicts during rebases.
2014
Git v2.0 Released v2.0
First major version bump. Cleans up old defaults: push.default changes to "simple," several deprecated options removed. Signals Git's maturity and commitment to stable, consistent behavior.
2015
Git LFS — Large File Storage
GitHub releases Git Large File Storage (LFS), extending Git to handle binary assets like images, videos, and ML models efficiently — replacing them with text pointers in the repo and storing the actual content on a separate server.
2018
Microsoft Acquires GitHub for $7.5B Industry
Announced June 4, 2018. Nat Friedman becomes CEO. Under Microsoft, GitHub launches Actions (CI/CD), Codespaces (cloud dev environments), Copilot (AI coding assistant), and Advanced Security — becoming a full DevOps platform.
2020
GitHub Renames Default Branch to "main"
GitHub changes the default branch name from "master" to "main" for all new repositories. Git 2.28 introduces the init.defaultBranch config option so users can set their own default.
2022
Git v2.36 — SHA-256 Repository Support
Git gains experimental support for SHA-256 object hashing — a long-awaited upgrade from SHA-1, which has known theoretical collision vulnerabilities. SHA-256 repos are not yet interoperable with SHA-1 remotes but the migration path is underway.
2023
Git v2.40 — Performance & UX Improvements
The 2.40.x series brings significant fetch performance improvements for monorepos, better partial clone support, improved git bisect with AI-assisted regression hunting, and incremental improvements to the bundle URI protocol for faster clones.
2025
Git Powers 420M+ Repositories Today
Git is the undisputed standard for version control. GitHub alone hosts over 420 million repositories. Git is installed on virtually every developer machine on the planet. What Linus hacked together in 10 days in 2005 is now as fundamental to software as the compiler itself.

Why Git?
🔀 Distributed by Design
Every developer has a full copy of the entire repository history. Work offline, commit locally, and sync when ready — no server required for day-to-day operations.
⚡ Blazing Fast
Branching, merging, and diffing happen in milliseconds because everything is local. Even massive repos with decades of history respond instantly to most operations.
🌿 Cheap, Powerful Branching
Creating a branch is just writing a 40-byte SHA reference. Experiment freely, maintain long-lived feature branches, and merge or rebase with confidence.
🔒 Data Integrity Built In
Every commit, file, and tree is cryptographically hashed (SHA-1/SHA-256). You cannot silently corrupt history — Git will detect any tampering or data loss immediately.
🕵️ Full Audit History
Every change ever made is recorded with author, timestamp, and message. git log, git blame, and git bisect let you trace any bug back to its origin commit.
🤝 Made for Collaboration
Pull requests, forks, and code review workflows are built around Git's model. Multiple developers can work on the same codebase simultaneously without stepping on each other.
↩️ Undo Anything
git revert, git reset, and git reflog mean you can recover from almost any mistake. Deleted a branch? Reflog has it. Bad merge? Revert it without touching history.
🏭 Industry Standard
~95% of professional developers use Git. Every CI/CD system, cloud platform, and IDE integrates with it. Learning Git means your skills transfer everywhere.
🧪 Safe Experimentation
Branch off, break things, and delete the branch. Your main codebase is untouched. Git's staging area also lets you craft precise, atomic commits from messy work-in-progress changes.
🚀 Enables CI/CD
Push triggers build. Merge triggers deploy. Git events are the heartbeat of modern DevOps pipelines — GitHub Actions, GitLab CI, and every major pipeline tool runs on Git hooks and webhooks.
📦 Not Just Code
Docs, configs, Terraform, Kubernetes manifests, Ansible playbooks, design files — if it's text, Git tracks it. Infrastructure as Code depends entirely on Git workflows.
🆓 Free & Open Source Forever
GPL v2 licensed, maintained by a global community, and will never be acquired, paywalled, or shut down. The tool you invest time learning today will exist in 20 years.
